## Codebase Patterns
- Persist the LAN server identity in `STORAGE_PATH/server_id` via `config.LoadOrCreateServerID()`; expose the exact same value from `/health` so clients can treat host/IP changes as the same server.
- Keep mDNS TXT metadata in lockstep with `/health` identity fields: when discovery clients need durable server identity, advertise the same `server_id` value in both places and cover it with a real `mdns.Query` test.
- For cross-platform LAN discovery iterations, write the behavior contract once in `tasks/lan-discovery-selection-contract.md` and mirror it in typed client models (`server_id` + `auto_discovered` / `restored_selection` / `manual_fallback`) before implementing per-platform restore logic.
- On macOS LAN config, discovered-server persistence must key selection/highlighting off `server_id` when available, and manual Host URL edits must clear stale discovered selection metadata back to `manual_fallback`.
- On macOS LAN config UI, render source badges and discovery guidance from `lanEndpointSource`, `selectedLanServer`, and the Bonjour result count instead of inferring state from `hostURL` alone; that keeps multi-server waiting and manual fallback visible.
- On Windows LAN config, route discovered selection, manual Host URL edits, and save-time normalization through `ConfigStore` helpers so `SelectedLanServer.ServerId` stays authoritative while `HostUrl` remains only the last-known/manual endpoint.
- On Windows startup restore, keep mDNS browsing running from launch but apply discovery-driven HostUrl rewrites only while `TransferMode == LanServer`; restore by `SelectedLanServer.ServerId`, and only auto-pick when discovery finds exactly one server and there is no saved/manual URL fallback.
- On Windows LAN config screens, derive discovery guidance from `LanEndpointSource`, `SelectedLanServer`, and current Bonjour result count, and expose an explicit manual-fallback action instead of making users edit `HostUrl` just to clear discovered selection state.
- On Android LAN config, persist discovered selection and manual Host URL edits through `ConfigStore` helpers so `StoredLanServerSelection.serverId` remains the durable match key, snake_case source values stay compatible on disk, and manual text edits clear stale discovered selection state before later restore/UI stories build on it.
- On Android LAN startup restore, keep `MdnsDiscoveryService` under `CopyEverywhereService` instead of the Config screen, expose discovered servers through a shared `StateFlow`, restore by `StoredLanServerSelection.serverId`, and only auto-select a lone discovered server when there is no saved selection and no manual Host URL fallback.
- On Android LAN config UI, drive source cards, manual-fallback actions, passive multi-server guidance, and discovered-server selection badges from `lanEndpointSource`, `selectedLanServer`, and the shared discovery list instead of inferring meaning from the Host URL text field.
- For DHCP-tolerant LAN fallback, persist the last manual Host URL separately from the active discovered endpoint on every client; when discovery misses a saved `server_id`, restore that manual URL and downgrade the source to `manual_fallback` instead of leaving the stale discovered address active.
- For LAN discovery diagnostics, log discovery start and resolved `server_id` events in the mDNS/Bonjour layer, but keep auto-select / restore / multi-server-wait / fallback-preserved logs at the selection-decision layer so one log stream explains both discovery inputs and restore outcomes.
