## Codebase Patterns
- Persist the LAN server identity in `STORAGE_PATH/server_id` via `config.LoadOrCreateServerID()`; expose the exact same value from `/health` so clients can treat host/IP changes as the same server.
- Keep mDNS TXT metadata in lockstep with `/health` identity fields: when discovery clients need durable server identity, advertise the same `server_id` value in both places and cover it with a real `mdns.Query` test.
- For cross-platform LAN discovery iterations, write the behavior contract once in `tasks/lan-discovery-selection-contract.md` and mirror it in typed client models (`server_id` + `auto_discovered` / `restored_selection` / `manual_fallback`) before implementing per-platform restore logic.
- On macOS LAN config, discovered-server persistence must key selection/highlighting off `server_id` when available, and manual Host URL edits must clear stale discovered selection metadata back to `manual_fallback`.
- On macOS LAN config UI, render source badges and discovery guidance from `lanEndpointSource`, `selectedLanServer`, and the Bonjour result count instead of inferring state from `hostURL` alone; that keeps multi-server waiting and manual fallback visible.
- On Windows LAN config, route discovered selection, manual Host URL edits, and save-time normalization through `ConfigStore` helpers so `SelectedLanServer.ServerId` stays authoritative while `HostUrl` remains only the last-known/manual endpoint.
- On Windows startup restore, keep mDNS browsing running from launch but apply discovery-driven HostUrl rewrites only while `TransferMode == LanServer`; restore by `SelectedLanServer.ServerId`, and only auto-pick when discovery finds exactly one server and there is no saved/manual URL fallback.
